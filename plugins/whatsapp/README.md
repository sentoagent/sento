# WhatsApp channel for Claude Code

Rides on your own WhatsApp number via Baileys. Inbound auto-replies on the
linked number; agent-initiated outbound to other contacts requires explicit
user confirmation.

**Status:** scaffolding only. Not yet functional.

## Heritage

Architecture cribs from [openclaw/openclaw `extensions/whatsapp/`](https://github.com/openclaw/openclaw/tree/main/extensions/whatsapp)
(MIT licensed). Patterns reused with attribution. We're explicitly **not**
copying code verbatim — we're learning from their architectural decisions:

- Baileys (`@whiskeysockets/baileys`) over whatsapp-web.js (no Chromium needed)
- Two-layer split: inbound-autonomous + outbound-confirmed
- JID/LID/E.164 identity normalization
- Reconnect with exponential backoff + jitter
- Group safety: membership × sender policy × mention gating

## Two-layer architecture

| Direction | Mode | Tool |
|---|---|---|
| User → agent on linked number | Auto-respond, allowlist gate | (built-in handler) |
| Agent → other contact | Confirmed outbound — show draft, wait approval | `send_whatsapp_to_contact` (confirm: true) |

The confirm pattern is what makes the companion-mode UX work: the agent can
only message your friends/family/clients with your explicit greenlight, and
the draft itself is generated by Claude in whatever language matches the
ongoing conversation.

## Auth

Two paths, mirroring OpenClaw:

- **QR code** — first run, Baileys emits `qr` event → render to terminal +
  base64 PNG for web setup. Best for local Mac installs.
- **Pairing code** — alternative for headless VPS where rendering a QR is
  awkward. User enters code in their WhatsApp app.

## State layout

```
~/.claude/channels/whatsapp/<accountId>/
├── .env              (no token — Baileys uses creds.json)
├── creds.json        (Baileys auth state, chmod 600)
├── creds.json.bak    (parse-before-clobber backup)
├── access.json       (allowFrom in E.164, dmPolicy, group rules)
└── inbox/            (pending inbound messages)
```

## Identity normalization (CRITICAL)

WhatsApp uses three different identifier formats:
- Classic JID: `5491134567890@s.whatsapp.net`
- LID: `5491134567890@lid` or `@hosted.lid` (newer system, gradual rollout)
- E.164 phone: `+5491134567890`

Multi-device adds a `:N` suffix to the JID. Always normalize before any
"is this from me?" check. See OpenClaw's [`identity.ts`](https://github.com/openclaw/openclaw/blob/main/extensions/whatsapp/src/identity.ts)
for the reference implementation — copy the shape, not the code.

## Reconnect policy

```ts
{ initialMs: 2_000, maxMs: 30_000, factor: 1.8, jitter: 0.25, maxAttempts: 12 }
```

On `DisconnectReason.loggedOut` → stop retrying, surface re-link prompt to
the user via Discord webhook (or whatever their primary channel is).

## Group safety

Three gates, must all pass for a group message to trigger the agent:

1. **Group membership allowlist** — is this group on the list?
2. **Sender policy** within allowed groups — is this sender authorized?
   (`open` / `allowlist` / `disabled`)
3. **Mention gating** — was the bot @mentioned or replied to?

Quote/reply alone does **not** grant authorization. (OpenClaw issue: classic
foot-gun where someone adds you to a group and starts spam-commanding.)

## Rate limiting (NEW — OpenClaw lacks this)

Two layers to reduce ban risk:

- Per-recipient: max 1 outbound per 10s per contact
- Per-account: max 30 outbound per minute total

WhatsApp's anti-spam heuristics flag rapid same-recipient or rapid
many-recipient patterns. These limits keep behavior in a "human typing on
phone" envelope.

## ToS warning

Baileys is **unofficial** WhatsApp client automation. It violates WhatsApp's
ToS. Meta can ban accounts using it. Per OpenClaw's blurb (worth copying):

> Works with your own number; recommend a separate phone + eSIM.

Sentō should surface this prominently in `npx sentoagent init` and the
website setup page when WhatsApp is selected.

## Open questions for next session

1. **Single-account vs multi-account** — OpenClaw supports multiple WhatsApp
   accounts per gateway (`<accountId>/` subdirs). Sentō v1 probably ships
   single-account; multi can come later. Worth noting in CLAUDE.md.

2. **QR rendering during init** — surface in the tmux session that's about
   to run the agent? Or pop a localhost webpage? Sentō currently has no
   localhost-webpage path; tmux QR is simpler.

3. **Embed vs separate-skill for outbound** — OpenClaw separates outbound
   into `wacli` (a Go binary). We can simplify to a single MCP server with
   confirm=true. Trade-off: simplicity vs blast-radius isolation.

4. **Session-expiry recovery** — when Baileys' creds.json expires (>14 days
   phone offline), Guardian needs to detect + notify. Likely a new log
   pattern added to Guardian's regex.
